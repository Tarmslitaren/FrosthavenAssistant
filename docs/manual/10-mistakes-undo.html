§10Mistakes, Undo, and Customization

§10.1Undo and redo

The mental model to hold for this section: the app draws a hard line between shared state and player preferences.

  • Shared state is the campaign — HP, conditions, initiatives, modifier draws, element imbues, scenario setup. Every modification is a command: it goes into a stream that syncs across all connected clients and can be undone.
  • Player preferences are your local settings — theme, scaling, language, which house-rule toggles your copy of the app uses. Preferences are written to local storage on the device that changed them. They never enter the command stream, never sync, and Undo never sees them.

Sidebar drawer → Undo reverses the last command in the shared stream. The drawer entry shows what would be undone (e.g., Undo: Imbue element fire), so you can confirm before tapping. Redo replays it. Because the stream is shared, an Undo on any connected client rolls the action back for everyone.

What gets undone (shared state — flows through the command stream):

  • HP / condition / XP changes on characters and monsters
  • Standee adds / removes
  • Element imbue / use
  • Initiative entries
  • Modifier deck draws
  • Round advances
  • Adding / removing characters and monsters
  • Setting a scenario / adding a section

What does not get undone (player preferences and local-only writes — never enter the command stream):

  • Settings. Toggling dark mode, scaling, language, or any other Settings menu item is a preference. It only changes the device that toggled it.
  • Network connection state. Connect/disconnect doesn't roll back through Undo.
  • Saved-character library. Loading or saving a character writes to local storage outside the command stream.

The undo stack is unbounded in practice (limited by available memory). After hundreds of commands in a session, you can still wind all the way back to scenario start — useful for "wait, what was the HP before that crit?" forensic reconstruction. Undo persists across saves, so closing and reopening the app preserves your history.

§10.2If something looks wrong with the data

The app's data files (monster ability decks, scenario rosters, class stats) are hand-curated from the rulebooks. With thousands of stat-card numbers and ability cards, occasional mistakes are inevitable.

If you notice a discrepancy:

  1. Confirm against the official rulebook or errata.
  2. File an issue at github.com/Tarmslitaren/FrosthavenAssistant/issues with the scenario number, monster type, and what's wrong.
  3. In the meantime, work around using Custom Difficulty (§4.5) — override the specific stat for the affected standee.

Data fixes are easy contributions. The JSON files are at frosthaven_assistant/assets/data/editions/<Edition>.json if you want to PR a fix yourself.

§10.3House rules

The app supports a few common house rules out of the box. The Scope column maps each rule onto the §10.1 distinction: Shared state rules ride the command stream and sync to every connected client (and are undoable); Local preference rules are per-device toggles that don't sync — every player at the table needs to set them on their own client if you want consistent behavior.

RuleSettingScopeWhen to use
Open initiative Settings → disable initiative secrecy (§9.6) Local preference Groups that don't care about hidden initiative reveals — initiatives become visible on your client as soon as they're entered.
Auto-expire conditions Settings → Auto-expire conditions Local preference Groups playing strict RAW: end-of-round Wound / Poison / Stun / Disarm / Immobilize / Muddle clear automatically. Turn it off if your table prefers to decide manually for edge cases.
Custom monster HP Long-press a stat card → set max HP Shared state One-off scenario tweaks — a monster's printed HP doesn't fit your house ruling. The change syncs and can be undone.
Custom scenario level Level widget on the bottom bar → difficulty offset (§4.5) Shared state Easy / Hard difficulty adjustments — applies to the whole scenario and to every connected client.
No standees Settings → No standees Local preference Groups that don't use physical standees — hides standee numbers throughout and tracks aggregate HP per type instead.

House rules that the app doesn't support directly (e.g., custom modifier-deck composition) typically need a manual workaround using the Add Monster + perks system, or a feature request.
